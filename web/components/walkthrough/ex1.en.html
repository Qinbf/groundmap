Should we adopt Graph RAG?

It depends on your query type — for global sensemaking / topic summarization the payoff is large; for simple factoid / single-hop QA it may not justify the index-time cost [[wiki/analyses/three_graph_rag_families#^p-6-942640]].

Is community detection required? (★ a real dispute — no single verdict)

  • GraphRAG (MSR): community detection + hierarchical summaries are the core architecture [[wiki/sources/graphrag#^h-2-3-c08b6a]]
  • LightRAG (HKU) refutes: community detection is unnecessary complexity; dual-level retrieval captures most of the benefit and supports incremental updates [[wiki/sources/lightrag]]
  • Knowledge-base verdict: the two engineering claims are real (added complexity + GraphRAG's lack of incremental updates), but the experimental numbers depend on LLM-as-judge so be cautious; status keep_watching [[wiki/concepts/graph_rag#^p-8-1775c9]]

The real disagreement is actually "expand-corpus school vs augment-retrieval school" — pick by your task spectrum (sensemaking vs full-spectrum QA), not by who's "better" [[wiki/analyses/three_graph_rag_families#^p-6-942640]].

⚠️ What classic chunk-RAG would do (argumentative comparison — not measured on this page)

This KB itself does not run chunk-RAG (design principle 1: the knowledge base doesn't call any LLM; principle 4: no embedding recall). What follows is a textbook-style projection of "what would go wrong" — unlike the rest of the page which is real captured trace data, this is argument.

1. Slice the GraphRAG / LightRAG / HippoRAG papers into 200–500 token chunks — dozens of fragments

2. Embedding-search top-K (usually K=5) — return 5 semantically similar chunks

3. Hand them to the LLM to synthesize — it will average "GraphRAG says community detection is core" and "LightRAG says it's unnecessary" into a bland "community detection is usually helpful" — the real architectural disagreement is flattened

4. Citation granularity = chunk (with surrounding-context noise) — the user can't trace back to which paper, which paragraph, let alone see a keep_watching "unresolved" status

5. Failure mode: conflict becomes invisible — the user gets a seemingly neutral, actually distorted answer, and misses the most important judgment: "this is an unresolved dispute — pick by query type"
