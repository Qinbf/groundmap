Agent 最终回复

要不要上 Graph RAG

取决于你的 query 类型——global sensemaking / 主题总结类收益大；简单 factoid / 单跳 QA 未必值回 index time 成本 [[wiki/analyses/three_graph_rag_families#^p-6-942640]]。

社区检测是必须的吗（★ 真实分歧，不给单一结论）

  • GraphRAG（MSR）：社区检测 + 层次摘要是核心架构 [[wiki/sources/graphrag#^h-2-3-c08b6a]]
  • LightRAG（HKU）反驳：社区检测是不必要的复杂度，dual-level retrieval 拿到大部分收益且支持增量更新 [[wiki/sources/lightrag]]
  • 知识库判断：两个工程论断真实（复杂度↑ + GraphRAG 不支持增量更新），但实验数字依赖 LLM-as-judge 需谨慎；状态 keep_watching [[wiki/concepts/graph_rag#^p-8-1775c9]]

更上层的综合

真正的分歧其实是"扩充 corpus 派 vs 辅助 retrieval 派"——选型应按你的任务谱（sensemaking vs 全谱 QA）来定，而非"谁更好" [[wiki/analyses/three_graph_rag_families#^p-6-942640]]。

⚠️ 传统 chunk-RAG 会怎样（论证性对比 · 不是本页实测）

本页 KB 本身不跑 chunk-RAG（设计原则 1:知识库不调用 LLM；原则 4:严禁 embedding 召回）。下面这段是按教科书级 chunk-RAG 范式推演的「会出什么问题」，与本页其它步骤不同——前者是真实跑出来的轨迹数据，后者是论证。

1. 把 GraphRAG / LightRAG / HippoRAG 几篇论文切成 200-500 token chunks → 大概几十个片段

2. 用 embedding 找 top-K（通常 K=5）→ 返回 5 个语义相似 chunk

3. 塞给 LLM 综合 → 它会把"GraphRAG 说社区检测是核心"和"LightRAG 说没必要"两类 chunk 平均成一句中庸的"社区检测通常有帮助"——真实的架构分歧被抹平

4. 引用粒度 = chunk（含上下文噪声）→ 用户无法反查到是哪篇论文的哪段，更看不到 keep_watching 这种"未决"状态

5. 失败模式：冲突不可见 —— 用户拿到一个看似中立、实则失真的答案，错过"这是一场未决之争、要按 query 类型选型"这个最关键的判断
