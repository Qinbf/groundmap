⚠️ 传统 chunk-RAG 会怎样

1. embedding 召回会把"TikTok Shop"和"Amazon"的相关 chunk 都拿到，但抓不到结构化对比维度（流量 vs 体量 vs 类目 vs 政策）

2. LLM 综合时倾向把所有数字平铺成一长段，失去"按场景分群"的决策结构

3. 无法表达信息不对称：用户不知道 Amazon 数据其实较弱、source_count = 0

4. 缺少"务实做法"维度——这种综合判断需要 wiki 的 analysis 页或 concept 页这种"二次综合"层，纯 chunk 检索很难给出

4 个例子分别证明了什么

例问题类型设计关键点 ★读了几个 wiki 页
1 事实 + 影响链路 分层聚合（source → concept → analysis）+ agent 不重复读底层 source 2 个（concept + analysis）
2 含数据冲突 entity 页表格直接列两个口径不挑边；analysis 页有完整冲突块 2 个（entity + analysis）

进阶：查询深度模式（quick · audit · explore · devil）

前面 4 个例子全是 quick 模式—— agent 读 2-4 个 wiki 页就给答案，这优化了"最快拿到答案"。 但 SMB 客户的高 stake 决策（合规 / 投资 / 估值 / 风险）需要更高强度的工作。 kb-query skill 设计了 4 个深度模式作为契约规范。

当前实施状态：Claude Code 中**默认且唯一**走 quick 模式（1-8 步流程）。 其他 3 个模式（audit / explore / devil）是**设计预留**—— 执行规范写在 SKILL.md 第 7a/b/c 步，等后续产品化定制开发时通过显式机制调用： k.py query --mode=audit CLI 参数 / HTTP API 的 "mode": "audit" 字段 / Web UI 下拉 / system prompt 注入。

为什么不做关键词自动判别

"深度模式"听起来像应该由 agent 自己识别——比如"用户说了'合规'就升 audit"。但实践上有 3 个 dealbreaker：

  • 语境模糊："合规"可能是问 De Minimis 合规细则（quick 够），也可能是 SMB 客户来确认产品合规策略（要 audit）——同一个词两种意图
  • 触发词膨胀：要覆盖所有 audit 场景，触发词表会越扩越大，最终全是高频词，等于全部升档，等于没用
  • 用户没掌控：agent 决定的模式，用户既不知道也不能否决，成本和延迟不可预测

正确做法：模式选择来自显式机制——产品化时由 UI 下拉 / API 字段 / CLI 参数明确传入。 这样模式行为对用户可见、可选、可预算。
